# Godot 4 游戏架构设计文档：InGame 与 OutGame 分离模式（无代码版）

## 1. 引言

在游戏开发中，将核心游戏逻辑（InGame）与用户界面交互（OutGame）清晰地分离，是保证代码可维护性、团队协作效率和项目扩展性的关键。本文档旨在提出一套基于 Godot 4 引擎的架构设计方案，通过合理的节点组织、状态管理和通信机制，实现 InGame 与 OutGame 的优雅解耦。

**目标**：
- 明确 InGame（游戏世界）和 OutGame（UI 界面）的职责边界。
- 提供可复用的节点结构和全局管理模块。
- 规范数据流和事件通信方式。
- 提升开发效率和项目可扩展性。

---

## 2. 整体架构概览

整个游戏以 **主场景** 为入口，内部划分为两大分支：**InGame 节点** 和 **OutGame 节点**。同时配合 **全局自动加载单例** 提供数据、事件和状态管理。

### 2.1 节点树结构

主场景下包含两个主要子节点：
- **InGame**：游戏世界根节点（Node2D/Node3D），所有实时玩法相关的节点（场景、角色、物理、摄像机等）均作为其子级。
- **OutGame**：UI 层，使用 `CanvasLayer` 作为根，确保 UI 始终显示在最前且不受游戏摄像机影响。可设置多个 `CanvasLayer` 以控制渲染层级（如背景层、普通界面层、弹窗层等）。

OutGame 下包含：
- **UIManager**：一个自定义控制节点，负责所有 UI 面板的打开、关闭、堆叠管理和焦点控制。
- **各个 UI 面板**：每个面板作为独立的场景（如主菜单、设置、暂停菜单、背包、HUD 等），在编辑器中作为 UIManager 的子节点放置。

### 2.2 全局自动加载单例

创建以下自动加载的单例，实现跨场景的数据共享和通信：
- **GameState**：游戏状态机，管理当前游戏模式（如 `PLAYING`、`PAUSED`、`IN_MENU` 等）。
- **PlayerData**：存储玩家属性（血量、金钱、背包等），不包含业务逻辑，仅作为数据容器。
- **GameEvents**：信号总线，任何模块均可在此发射和监听信号，实现低耦合通信。

## 3. 模块详细设计

### 3.1 InGame 模块

**职责**：
- 维护游戏世界的运行（物理更新、AI、玩家控制）。
- 与 OutGame 共享数据（通过 PlayerData）并触发状态变化（通过 GameState）。
- 不直接引用具体的 UI 控件，仅通过信号或全局事件间接影响 UI。

**设计要点**：
- 所有实时逻辑应放在 InGame 及其子节点中。
- 摄像机控制、场景切换等均在此处理。
- 当游戏暂停（GameState 进入 PAUSED）时，InGame 应停止物理更新（可通过设置节点的处理模式或代码逻辑实现）。

### 3.2 OutGame 模块

**职责**：
- 显示所有用户界面，包括菜单、HUD、弹窗等。
- 响应用户输入，将操作转换为对全局数据或事件的调用。
- 管理 UI 面板的打开/关闭、焦点、动画过渡。

#### 3.2.1 UIManager 设计

UIManager 是 OutGame 的核心控制器，负责所有面板的生命周期管理。主要功能包括：
- 自动注册所有子面板（通过节点名称或预定义的标识符）。
- 提供打开/关闭面板的方法，支持传递初始化数据。
- 维护面板堆栈，处理层叠弹窗的显示与关闭。
- 根据面板属性（如是否独占、是否暂停游戏）控制游戏状态和输入焦点。
- 统一管理面板的动画过渡（如淡入淡出、缩放）。

每个 UI 面板应具备以下特性：
- 独立的场景文件，便于开发和测试。
- 可选的初始化方法，用于接收外部传入的数据。
- 可配置的属性（如 `pauses_game` 布尔值）告知 UIManager 该面板是否需要暂停游戏。

### 3.3 游戏状态机（GameState）

游戏状态机作为全局单例，维护当前游戏状态。状态变更时发出信号，各模块监听并做出相应调整。常见状态包括：
- `PLAYING`：正常游玩，InGame 更新，OutGame 显示 HUD。
- `PAUSED`：游戏暂停，显示暂停菜单，InGame 停止更新。
- `IN_MENU`：处于主菜单或其他纯 UI 状态，InGame 可能未加载。
- `GAME_OVER`：游戏结束，显示结束界面。

### 3.4 全局数据与事件

**PlayerData** 作为数据容器，存储玩家相关数据，并提供数据变更信号（如金币变化、背包更新）。数据修改应通过 PlayerData 提供的方法进行，以便发射信号。

**GameEvents** 作为全局信号总线，定义游戏中可能发生的各类事件（如道具拾取、敌人死亡、对话开始等）。任何模块均可发射或连接这些信号，实现解耦的跨模块通信。

---

## 4. 输入处理策略

为避免 UI 输入穿透到游戏世界，采用状态驱动的输入处理。

- **InGame 输入**：在 InGame 的输入处理中，首先检查当前游戏状态。仅在 `PLAYING` 状态下处理游戏内输入（如移动、攻击），否则忽略输入。
- **OutGame 输入**：UI 控件自动处理自身的输入。UIManager 可捕获全局返回键（如 ESC），根据面板堆栈决定是关闭顶层菜单还是打开暂停菜单。
- **CanvasLayer 阻断**：可启用 CanvasLayer 的阻断输入属性，使所有 UI 下的输入都被拦截，但这种方法较粗粒度，建议仅在必要时使用。

---

## 5. 场景切换与动态加载

当从 OutGame（如主菜单）进入 InGame 时，需要动态加载游戏世界场景。

**流程**：
1. 主场景中的 InGame 节点作为占位符，初始可能为空。
2. 调用加载方法，使用 Godot 的资源加载系统（建议异步加载）实例化世界场景。
3. 将实例化的世界添加到 InGame 节点下。
4. 通过 UIManager 切换到 HUD 面板，并更新游戏状态为 `PLAYING`。
5. 加载期间可显示加载界面（作为 UI 面板之一），提升用户体验。

## 6. 通信机制总结

| 通信类型         | 实现方式                     | 描述                                     |
|------------------|------------------------------|------------------------------------------|
| InGame → OutGame | 通过 GameEvents 或 PlayerData 信号 | 游戏内事件触发 UI 更新，如拾取道具后刷新背包显示。 |
| OutGame → InGame | 通过 GameState 或直接调用 PlayerData | 用户操作（如使用道具）修改数据，InGame 监听数据变化做出反应。 |
| 跨模块数据共享   | 全局单例                     | 所有模块均可读写 PlayerData 等全局数据。 |
| UI 内部通信      | 直接调用或信号               | 按钮点击触发 UIManager 打开/关闭面板。   |

## 7. 实现注意事项（无代码）

- **面板独立性**：每个 UI 面板应尽量少依赖外部全局变量，通过初始化方法接收必要数据。
- **主题统一**：为 OutGame 设置全局主题，统一字体、颜色、样式，提高开发效率。
- **动画过渡**：面板打开/关闭可配合动画播放器实现平滑效果，建议在 UIManager 中统一控制。
- **调试便利**：开发期间可保留快捷键直接打开特定面板。
- **性能优化**：大量 UI 控件应考虑复用或使用轻量级容器；SubViewport 渲染较耗资源，仅在必要时使用。
- **输入设备适配**：确保 UI 支持键盘/手柄导航，利用焦点邻接等属性。

---

## 8. 扩展性考虑

- **多 UI 层级**：可设置多个 CanvasLayer（如背景层、普通界面层、弹窗层），通过层属性控制渲染顺序。
- **插件化 UI**：将 UIManager 做成插件，便于在不同项目中复用。
- **游戏模式扩展**：状态机可轻松添加新状态（如过场动画、小游戏模式）。
- **网络同步**：若游戏支持多人，PlayerData 可扩展为网络同步的数据容器。

---

## 9. 结语

本文档描述的 InGame/OutGame 分离架构充分利用了 Godot 4 的节点树、信号和自动加载机制，实现了清晰的职责划分和高效的开发流程。开发者可根据实际项目需求调整细节，但核心原则——解耦、模块化、数据驱动——值得长期坚持。
